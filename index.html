The problem

The protocols are multiplying. The identity layer above them isn't.

Today, agents borrow financial identity from humans — shared API keys, shared cards, shared billing accounts. Five ways that breaks down.

01 / 05

Agents hold raw credentials

Developers embed real card numbers and API keys in agent context. A single prompt injection or context leak exposes the actual account — with no per-agent cap and no way to revoke just one agent.

02 / 05

Six protocols, six silos

ACP, AP2, TAP, Agent Pay, KYAPay, x402 — each one solves a slice of the execution layer with its own registration and credential format. There is no unified identity above any of them.

03 / 05

Budget checks happen after the charge

Payment networks authorise after the transaction is submitted. Concurrent agents race each other and collectively overspend before any check fires.

04 / 05

LLM spend and purchase spend live in different systems

An agent can be well within its Claude budget while dramatically overspending on purchases. No single view of what an agent actually cost you.

05 / 05

A bad agent on one rail looks fine on the next

Reputation doesn't follow the agent. A behavioural profile built on one rail is invisible on every other, so merchants can't screen agents before fulfilling a high-value order.

"Every financial system built in the last fifty years assumes a human is the one spending. Agents aren't humans. They need a financial identity of their own."
